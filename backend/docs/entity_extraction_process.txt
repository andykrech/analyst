Процесс извлечения сущностей (tech, person, phenomenon) из квантов
==================================================================

1. Выборка квантов для обработки
--------------------------------

- В таблице theme_quanta выбираются кванты по теме с `entity_extraction_version IS NULL`.
- Кванты берутся батчами, упорядоченными по `created_at`, с блокировкой `FOR UPDATE SKIP LOCKED`,
  чтобы параллельные воркеры не брали одни и те же строки.

2. Извлечение технологий (entity_type = "tech")
-----------------------------------------------

- Для каждого кванта строится текст из `title` + `summary_text`.
- Текст отправляется в LLM по промпту "entities_extractor_tech".
- Ответ парсится в список кандидатов технологий.
- Все кандидаты из батча группируются по ключу (theme_id, normalized_name):
  - накапливается множество quantum_ids, в которых встречается технология;
  - собираются варианты канонических имен (canonical_name) для голосования;
  - вычисляются min/max даты появления по кванту.
- Для каждой группы:
  - если сущность ещё не существует, создаётся запись в `entities` с:
    - `entity_type = 'tech'`,
    - `normalized_name` (ключ),
    - `canonical_name` (с учётом перевода на основной язык темы, если нужно),
    - `mention_count`, `first_seen_at`, `last_seen_at`;
  - если сущность существует, обновляются агрегаты (mention_count, first/last_seen_at),
    при необходимости обновляется canonical_name.
- Для всех технологий создаются связи в таблице `relations`:
  - по одной записи на пару (entity_id, quantum_id) с:
    - `subject_type = 'entity'`,
    - `object_type = 'quantum'`,
    - `relation_type = 'mentions'`,
    - `status = 'active'`.

3. Извлечение персон (entity_type = "person")
---------------------------------------------

- По кванту собираются кандидаты авторов публикации (person_publication_authors_extractor):
  - извлекаются имена авторов, нормализуются варианты написания;
  - кандидаты группируются по нормализованному имени.
- Группы кандидатов коллапсируются и применяются к БД:
  - создаются/находятся сущности `entities` с `entity_type = 'person'`;
  - для каждой найденной персоны и кванта создаётся связь в `relations`:
    - `subject_type = 'entity'`,
    - `object_type = 'quantum'`,
    - `relation_type = 'author'`,
    - `status = 'active'`.
- Эти же связи (`relation_type = 'author'`) затем используются в экспортном скрипте
  для вывода персон по каждому квантy.

4. Извлечение явлений (entity_type = "phenomenon")
--------------------------------------------------

4.1. Вызов LLM-экстрактора

- Для каждого кванта строится текст из `title` + `summary_text`.
- Текст отправляется в LLM по промпту `entity.entities_extractor_phenomenon.v1`.
- Ответ парсится в модель:
  - список явлений с полями:
    - phenomenon (само явление на языке кванта),
    - modifier (up / down / stable / unstable / none),
    - condition_text (строка с условием/контекстом).
- Модификаторы, не входящие в допустимый список, приводятся к `"none"`.

4.2. Группировка явлений

- Для каждого кванта и каждого кандидата явления:
  - нормализуется имя явления (переводится в `normalized_name`, как правило на английский);
  - строится ключ группы: (theme_id, normalized_name).
- Для каждой группы накапливается структура:
  - `normalized_name`,
  - множество `quantum_ids`, где явление встретилось,
  - список вхождений:
    - для каждого вхождения: (quantum_id, modifier, condition_text),
  - набор surface-форм (как явление было написано в тексте),
  - минимальная и максимальная дата появления по квантам.

4.3. Создание/обновление сущностей-явлений

- Для всех групп загружаются уже существующие сущности `entities` с `entity_type = 'phenomenon'`.
- Для каждой группы:
  - если сущность ещё не существует или у неё пустой canonical_name:
    - определяется основной язык темы (primary language);
    - при необходимости вызывается перевод `normalized_name` с английского
      на язык темы;
    - canonical_name строится так, чтобы только первая буква была заглавной,
      остальные строчные (специально для явлений);
  - создаётся новая запись `entities` или обновляются поля существующей
    (mention_count, first_seen_at, last_seen_at, canonical_name, is_name_translated).

4.4. Алиасы явлений

- Для каждой группы и связанной сущности:
  - берутся surface-формы явления;
  - если форма отличается от normalized_name и canonical_name (по нижнему регистру),
    создаётся запись в `entity_aliases`:
    - `entity_type = 'phenomenon'`,
    - `alias_value` = surface-форма,
    - `kind = 'surface'`,
    - `source = 'ai'`.

4.5. Связи и утверждения (Relation + RelationClaim)

- Для каждой группы явлений и каждого кванта, где оно встретилось,
  создаётся связь в `relations`:
  - `subject_type = 'entity'`,
  - `subject_id` = id сущности-явления,
  - `object_type = 'quantum'`,
  - `object_id` = id кванта,
  - `relation_type = 'mentions'`,
  - `status = 'active'`.
- Для каждого вхождения явления (quantum_id, modifier, condition_text):
  - ищется соответствующая связь entity–quantum;
  - в таблицу `relation_claims` добавляется (или не дублируется, если уже есть)
    запись:
    - `relation_id` = id связи,
    - `property_type = 'phenomenon_modifier_condition'`,
    - `properties_json = {"modifier": ..., "condition_text": ...}`.

5. Обновление версии извлечения
-------------------------------

- После успешной обработки батча квантов для всех трёх типов сущностей
  (tech, person, phenomenon) в таблице `theme_quanta` для этих квантов
  заполняется поле `entity_extraction_version` текущим значением из настроек
  (например "v1.0") и обновляется `updated_at`.

6. Экспорт для анализа (backend/docs/quanta_entities.txt)
---------------------------------------------------------

- Отдельный служебный скрипт `app.scripts.export_quanta_entities`:
  - выбирает все кванты из `theme_quanta`;
  - подтягивает связанные сущности через `relations`:
    - технологии (`entity_type='tech'`, `relation_type='mentions'`);
    - персоны (`entity_type='person'`, `relation_type='author'`);
    - явления (`entity_type='phenomenon'`, `relation_type='mentions'`);
  - для явлений дополнительно подтягивает `relation_claims` с
    `property_type='phenomenon_modifier_condition'` и выводит модификатор
    и условие по каждой связке сущность–квант;
  - пишет всю информацию в файл `backend/docs/quanta_entities.txt`
    в удобочитаемом текстовом формате для ручного анализа.
